## Problem Statement

现有项目是一个功能完整的 demo：基于 LangGraph + DeepAgents SDK，主 Agent 自主规划任务，委派给三个子 Agent（网络搜索、数据库查询、RAGFlow 知识库），最终生成 Markdown/PDF 报告。功能能跑，但代码质量停留在原型级别——代码结构暴露出"照着教程写的"痕迹，而非"经过设计思考的"。

## 目标

把这个项目改造为一个**架构清晰、可完整讲述设计决策的**AI Agent 项目。核心判断标准：

1. **git log 能展示设计思考的演进**——每个 commit 都是一个设计决策
2. **README 能展示架构能力**——不只是"怎么跑起来"，而是"为什么这么设计"
3. **代码每个模块都能讲出"为什么这么写"**——不是"照着文档写的"
4. **逐步迭代**——不是一次性做完，而是分阶段推进，每个阶段都能独立讲出一个故事

## 当前代码的致命问题（生产环境必须修复的）

### 1. 子 Agent 只是 dict，没有架构故事

```python
# 当前代码
network_search_agent = {
    "name": sub_agents_config["tavily"].get("name", ""),
    "description": ...,
    "system_prompt": ...,
    "tools": [internet_search]
}
```

三个子 Agent 就是三个 dict 字面量。审查代码的人会问："你的 Agent 之间怎么共享上下文？怎么做状态管理？"——当前代码答不上来，因为没有这个设计。

### 2. 工具函数混合了太多职责

`mysql_tools.py` 里每次调用工具都 `load_dotenv()`，`tavily_tools.py` 在模块顶层创建 client 但没有错误降级。工具的职责应该是"执行业务逻辑"，但当前代码里 env 加载、错误处理、监控上报和业务逻辑耦合在一起。

### 3. SQL 注入风险（P0）

```python
safe_table_name = table_name.replace("`", "").replace(";", "").split()[0]
cursor.execute(f"select * from {safe_table_name} limit 100")
```

这是黑名单过滤，不是白名单校验。如果 table_name 是 `users; DROP TABLE`，当前代码无法阻止，这暴露了安全意识不足。

更严重的是 `execute_sql_query` 直接执行任意 SQL，没有限制为只读查询。`DROP TABLE`、`DELETE FROM` 等破坏性操作都能执行。

### 4. 零测试

没有任何 unit test 或 integration test。对于 AI Agent 项目，测试是展示工程素养的最直接方式。

### 5. Agent 之间没有显式通信

三个子 Agent 各自独立工作，结果汇总给主 Agent。没有"网络搜索结果可以指导数据库查询"这样的协作模式。这在 LangGraph 里是完全可以通过 Shared State 实现的。

### 6. 安全隐患（独立评审补充）

- **文件上传路径遍历**：`api/server.py:59` 直接使用 `file.filename` 拼接路径，`../../../etc/passwd` 可逃出目标目录
- **CORS 全开放**：`api/server.py:30` 设置 `allow_origins=["*"]`
- **asyncio.create_task fire-and-forget**：`api/server.py:47` 不保存 task 引用，异常静默丢失

### 7. 跨平台兼容性

- **Word-to-PDF 仅限 Windows**：`utils/word_converter.py:50-51` 使用 `pywin32` 和 `win32com.client`，在 Linux Docker 容器中直接报错

## 保留的亮点

- LangGraph + `create_deep_agent` 的用法是对的
- ContextVar 做异步请求隔离的设计合理
- YAML 外部化 prompt 配置
- WebSocket 实时流的架构方向正确

## 改造方案（7 个阶段，逐步迭代，每个阶段对应一组 commit）

### Phase 0: 安全基线（最紧急的修复，1-2 个 commit）

**问题**：SQL 注入、路径遍历、CORS 全开——这些是安全漏洞，不是功能问题。修好它们，可以说"我在重构的第一步就发现了安全问题并立即修复"。

**改动**：
- `tools/mysql_tools.py` — `execute_sql_query` 限制为只读（只允许 SELECT），表名校验改为白名单（从 `SHOW TABLES` 获取合法表名列表）
- `api/server.py` — 文件上传路径遍历修复：`Path(file.filename).name` 提取纯文件名；CORS 限制到前端源
- `api/server.py` — `asyncio.create_task` 保存引用 + 添加异常回调

**设计意图**："我在 review 代码时发现 SQL 查询可以执行任意语句，包括 DROP TABLE。虽然 Agent 的 prompt 说只让它查询，但安全不能依赖 prompt。我立刻加了白名单校验和只读限制。"

### Phase 1: 子 Agent 架构升级

**问题**: 子 Agent 是 dict，没有状态、没有记忆、没有上下文共享。

**设计**: 引入 AgentConfig TypedDict + AgentContext 模式。

```python
@dataclass
class AgentContext:
    """Agent 执行上下文，支持跨工具调用的状态共享"""
    thread_id: str
    workspace_dir: Path
    session_dir: str  # 从 api.context 整合，避免两个并行状态系统
    memory: dict[str, Any]  # 短期记忆：本次会话的中间结果
    metadata: dict[str, Any]  # 元数据：调用次数、token 消耗等
```

每个子 Agent 从 dict 升级为一个带类型注解的配置对象，附带 AgentContext 用于状态追踪。

**注意**：
- `api/context.py` 已有 `get_session_context()` 和 `get_thread_context()`，AgentContext 应整合这些值，而不是搞两套状态
- `deepagents.create_deep_agent` 期望 subagents 是特定格式的 dict，AgentConfig 必须兼容这个接口。如果不完全兼容，需要在 `main_agent.py` 里做一层适配
- `monitor` 单例被所有工具模块导入，这是紧耦合。理想做法是通过 AgentContext 注入，但第一阶段可以先保持现状，Phase 4 再解耦

**改动文件**:
- `agent/sub_agents/base.py` — AgentContext + AgentConfig 基类
- `agent/sub_agents/network_search_agent.py` — 重构
- `agent/sub_agents/database_query_agent.py` — 重构
- `agent/sub_agents/knowledge_base_agent.py` — 重构

### Phase 2: 工具工程化

**问题**: 工具函数混合 env 加载、监控、业务逻辑。MySQL 每次新建连接。tavily 工具抛异常而其他工具返回错误字符串。

**设计**: 引入连接管理器和关注点分离。

```python
class MySQLConnectionManager:
    """管理数据库连接生命周期，支持连接池和超时"""
    def __init__(self, config: DBConfig):
        self._pool = mysql.connector.pooling.MySQLConnectionPool(...)

    def execute_query(self, sql: str, params: tuple = None) -> QueryResult:
        ...
```

**关键决策**：
- `mysql.connector` 的 pooling 是同步的，在 async 上下文会阻塞事件循环
- 两个方案：(a) 用 `mysql.connector` 的同步 pool + `asyncio.to_thread` 包装；(b) 换 `aiomysql` 异步驱动
- **推荐 (a)**：改动最小，不需要改驱动，可以说"我发现了同步阻塞问题，用 asyncio.to_thread 解决了"
- 如果想更激进，选 (b)：但需要确认 `aiomysql` 和现有 SQL 兼容性

**改动文件**:
- `tools/db_connection.py` — 新增连接管理器（异步安全）
- `tools/mysql_tools.py` — 重构（使用 ConnectionManager）
- `tools/ragflow_client.py` — 新增 RAGFlow client 封装（连接复用 + session 泄漏修复）
- `tools/ragflow_tools.py` — 重构
- `tools/tavily_tools.py` — 重构（重试 + 超时 + 改为返回错误字符串而非抛异常）
- 统一所有工具的 env 加载：移到 `api/server.py` 启动时，工具只读 `os.environ`

### Phase 3: Agent 间通信（简化方案）

**问题**: 子 Agent 之间没有信息共享。主 Agent 拿到所有结果后手动拼接。

**设计决策**：
- 原方案设计了独立的 SharedBus（发布/订阅），但独立评审指出 LangGraph 的 StateGraph 已经提供了节点间共享状态
- **简化方案**：利用 LangGraph 的 StateGraph 共享状态，加一个薄封装让 Agent 可以"发布事实"和"查询事实"
- 这样可以说："我最初设计了独立的 pub/sub 总线，后来发现 LangGraph 的 StateGraph 已经提供了这个能力。我做了简化，在框架之上加了一个语义层"

```python
class SharedContext:
    """基于 LangGraph StateGraph 的轻量级共享上下文"""
    # 不是独立的 pub/sub，而是 LangGraph state 的语义封装
    def publish_fact(self, state, fact: str, source: str, topic: str) -> str:
        """发布事实到 state"""
        ...
    def query_facts(self, state, topic: str) -> list[str]:
        """按主题查询事实"""
        ...
```

**改动文件**:
- `agent/shared_context.py` — 新增（基于 LangGraph state 的薄封装）
- `agent/main_agent.py` — 改造（StateGraph 中加入 shared_facts 字段）
- `agent/sub_agents/*.py` — 改造（使用 SharedContext 发布/查询事实）

### Phase 4: 可观测性

**问题**: 只能看到"工具开始/结束"，看不到调用链、token 消耗、性能瓶颈。Monitor 日志输出完整参数（可能泄露凭证/PII）。

**设计**: 引入 AgentTelemetry 收集关键指标 + 日志脱敏。

```python
@dataclass
class AgentTelemetry:
    thread_id: str
    agent_name: str
    tool_name: str
    duration_ms: float
    input_tokens: int
    output_tokens: int
    status: str  # success | error | timeout
    error: str | None
```

**改动文件**:
- `agent/telemetry.py` — 新增
- `api/monitor.py` — 改造（集成 telemetry 收集 + 日志参数脱敏）
- `api/server.py` — 新增 `/api/telemetry` 端点

### Phase 5: 测试

**问题**: 零测试。

**设计**: 分层的测试策略。

```
tests/
├── unit/
│   ├── test_mysql_connection.py      — 连接管理器 + 并发连接池行为
│   ├── test_mysql_tools.py           — 表名白名单校验（SQL 注入防御）
│   ├── test_path_utils.py            — 路径解析 + 路径遍历防御
│   ├── test_shared_context.py        — 共享上下文发布/查询
│   └── test_server.py                — 文件上传路径遍历、CORS、task 异常处理
├── integration/
│   ├── test_agent_delegation.py      — Agent 委派链路（mock LLM 响应）
│   ├── test_report_generation.py     — 报告生成（验证文件生成到正确目录）
│   └── test_context_isolation.py     — 两个并发 run_deep_agent 调用，验证 ContextVar 隔离
└── conftest.py                        — 共享 fixtures
```

**关键决策**：
- 集成测试需要 mock LLM 响应（LLM 输出不确定，不能每次都调真实 API）
- 单元测试覆盖工具函数的确定性逻辑（白名单校验、路径解析等）
- ContextVar 隔离测试：两个并发请求，验证 session_dir 不交叉

**设计意图**："给 AI Agent 写测试和普通代码不一样。LLM 的输出是不确定的，所以我把测试分成两层：工具函数写单元测试（确定性逻辑），Agent 调用链写集成测试（验证结构而非具体输出）。最难的是 ContextVar 隔离测试——我写了两个并发请求，验证它们的工作目录不会交叉。"

### Phase 6: 跨平台 PDF + 部署

**问题**: 没有 Docker，没有一键部署。Word-to-PDF 仅限 Windows。

**设计变更**：
- PDF 引擎从 `pywin32` 替换为跨平台方案
- 两个选择：(a) **pandoc** — 命令行工具，跨平台，支持 MD → PDF 直接转换；(b) **WeasyPrint** — Python 库，MD → HTML → PDF
- **推荐 (a) pandoc**：不引入 Python 依赖，Dockerfile 里 `apt install pandoc` 即可，可以说"我发现了 Windows-only 的依赖，换成了跨平台的 pandoc"
- 如果选 (b) WeasyPrint：纯 Python，但需要安装系统字体依赖

**改动文件**:
- `utils/pdf_converter.py` — 新增（基于 pandoc 的跨平台 PDF 转换）
- `tools/pdf_tools.py` — 重构（使用新 converter）
- `Dockerfile` — backend 镜像（Python + pandoc）
- `Dockerfile.frontend` — frontend 镜像（Node.js）
- `docker-compose.yml` — 编排 backend + frontend + MySQL
- `Makefile` — 常用命令封装

## 技术选型说明

| 决策 | 选择 | 为什么 | 备选 |
|------|------|--------|------|
| Agent 间通信 | LangGraph StateGraph + 薄封装 | 不重复造轮子，简化方案更好讲 | 独立 SharedBus（太重，和框架重复） |
| 测试框架 | pytest | Python 生态标准 | unittest（太底层） |
| 数据库连接池 | mysql.connector pooling + asyncio.to_thread | 官方驱动内置，改动最小 | aiomysql（需要换驱动，兼容性不确定） |
| 可观测性 | 自定义 telemetry | 展示设计能力，不依赖第三方 | LangSmith（黑盒，讲不出设计） |
| PDF 引擎 | pandoc | 跨平台，Docker 友好，一行命令 | WeasyPrint（纯 Python 但需字体依赖） |
| SQL 安全 | 白名单 + 只读限制 | 彻底杜绝注入 | 黑名单过滤（当前方案，已被绕过） |

## 成功标准

1. **git log 至少 20 个有意义的 commit**，每个 commit message 能说清楚"改了什么、为什么改"
2. **README 包含架构图、设计决策、技术选型说明**，不只是"怎么跑"
3. **测试覆盖率 > 60%**（工具函数 + 核心逻辑）
4. **能完整讲出以下故事**：
   - 为什么从安全漏洞开始修（Phase 0 的故事）
   - 为什么从 dict 升级到 AgentConfig（Phase 1）
   - 同步连接池阻塞事件循环怎么解决的（Phase 2）
   - SharedBus 怎么简化为 LangGraph StateGraph 封装的（Phase 3 的迭代故事）
   - 可观测性发现了什么性能问题（Phase 4）
   - 给 AI Agent 写测试有什么特殊挑战（Phase 5）
   - Windows-only 依赖怎么替换的（Phase 6）
