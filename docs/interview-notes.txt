面试官您好，我想跟您详细介绍一下我最近做的一个**“基于 DeepAgents 的企业级深度搜索与研究智能体”**项目。
其实做这个项目的初衷，是因为我们发现企业在做复杂决策的时候，信息太分散了。
第一，先说说这个项目在企业里的价值。
您知道，传统那种 RAG 系统，往往只能单次检索，碰到那种需要跨来源、多步骤的复杂任务，它就搞不定了。比如领导想看一个行业分析报告，以前可能需要人工去搜公开新闻、查内部数据库、再翻私有文档，最后还要写报告，耗时很长。
我们这个系统，其实就是构建了一个“深度搜索研究员”。它能自动把公开互联网信息、企业内部的 MySQL 数据，还有私有知识库（比如 RAGFlow）打通。原本需要人工几小时完成的“搜索 - 整理 - 撰写”流程，现在能自动化变成分钟级输出，而且支持直接生成 Markdown 或者 PDF 报告。另外，我们还特别加了人机协作机制，涉及到删库、删文件这种高危操作，必须人工审批，这就满足了企业的合规安全要求。
第二，关于项目架构。
我们没用那种死板的工作流，而是采用了**“主智能体统筹 + 多专家子智能体并行”**的架构。
您可以想象成一个“项目经理”带着三个“专家助手”。
主智能体负责理解需求、拆解任务；
然后它会把任务分发给三个子智能体：一个是负责外网搜索的，一个是查内部 MySQL 数据库的，还有一个是查私有知识库的。
技术上，后端我们用 FastAPI 配合 WebSocket 实现异步服务，核心编排是基于 DeepAgents 和 LangGraph 构建的状态化工作流。这样既能保证多用户并发时的隔离，又能让智能体具备记忆和规划能力。
第三，这个项目的亮点。
我觉得最有意思的有三点：
它会自己规划。 基于 DeepAgents 的 write_todos 机制，智能体接到任务不是瞎忙活，而是先列待办清单。执行中发现搜索不到信息，它会自动调整计划，比如换个关键词再搜，具备闭环纠错能力。
解决了上下文溢出问题。 长任务容易导致 Context 不够用，我们设计了“文件系统卸载机制”。智能体把中间结果写到一个虚拟文件系统里，按需读取，这样就能支撑很长链路的任务。
全链路可观测。 Agent 执行过程通常是黑盒，但我们自研了一套监控中间件，通过 WebSocket 把智能体每一步的思考、工具调用、耗时都实时推送到前端。这样不仅用户体验好，我们调试起来也方便。
第四，开发过程中遇到的难点。
确实有几个坑比较深：
异步并发下的会话隔离。 因为 FastAPI 是异步的，多用户请求在同一个线程里跑，如果用全局变量，用户数据就会串线。我当时引入了 Python 的 ContextVars 上下文变量，为每个请求链路绑定独立的 Thread ID 和文件路径，确保深层工具调用时数据不串扰。
Agent 幻觉问题。 比如查数据库，模型可能会臆造表名。我设计了一个“三步走”策略：强制它先查表结构，再预览数据，最后才能写 SQL。同时在工具层做了安全清洗，防止 SQL 注入。
复杂文件路径管理。 跨平台的路径分隔符很容易出错。我封装了一个统一的路径解析工具，基于 pathlib，自动清洗虚拟路径前缀，防止路径越权访问。
第五，我在项目里承担的工作。
我是作为核心开发工程师参与的， basically 负责了从架构到落地的全流程：
架构选型上，我确定用 DeepAgents 框架，设计了"1+N"的多智能体协作模式。
智能体开发上，我编写了主智能体和三个子智能体的 System Prompt，通过 Few-Shot 和思维链优化了任务拆解的准确率。
工具链这块，我独立开发了 MySQL 数据库工具、搜索工具以及文件转换工具。
后端服务也是我搭建的，包括 FastAPI 服务、WebSocket 消息推送，还有刚才提到的基于 ContextVars 的会话隔离机制。
最后，再补充一点我个人的思考。
我觉得当前大模型应用正从“对话式”向“代理式”转变。在这个项目里，我特别注重状态管理和可控性。
比如在生产环境，成本很重要，我未来计划引入小模型路由，简单任务用小模型，复杂任务再用大模型。
另外，安全是底线，我在数据库工具和文件操作中都做了防御性编程，并设计了 HITL 审批流。
我认为，能让 Agent 在企业里真正落地的，不仅仅是 Prompt 写得好，更是工程化的稳定性、可观测性和安全性。
以上就是我这个项目的介绍，谢谢面试官。
