# Decision Research Agent v0.1.0

## Supported Surface

v0.1.0 is a backend-and-CLI release. It includes:

- FastAPI backend service;
- canonical `POST /api/runs` and `GET /api/runs/{run_id}/result` APIs;
- Python Tool Client at `tools/decision_research_agent_tool.py`;
- DeepAgents-native generic research harness;
- bounded Talent Hiring Signal profile;
- controlled durable review and evidence verification workflows behind
  disabled-by-default flags;
- operator scripts, tests, specifications, and documentation.

No frontend service is shipped. React deferred to a later product slice that
must consume the canonical API and result contracts.

Delivery is Markdown-only delivery. The release does not generate PDF files.

## Breaking Changes

- Pre-v0.1.0 compatibility aliases and task/thread routes were removed from the
  active runtime.
- Pre-v0.1.0 Tool Client shims were removed; use
  `tools/decision_research_agent_tool.py`.
- File upload/download and in-agent report conversion are not part of this
  release.
- Historical tables may still exist in old databases, but the runtime uses
  canonical run, artifact, review, verification, and publication tables.

## Migration

1. Create a local environment file and set canonical environment variables:

   ```bash
   cp .env.example .env
   ```

   Required persistent database variables:

   ```dotenv
   DECISION_RESEARCH_AGENT_DB_PATH=data/decision_research_agent.db
   DECISION_RESEARCH_AGENT_CHECKPOINT_DB_PATH=data/review_checkpoints.db
   DECISION_RESEARCH_AGENT_ENABLE_DURABLE_HITL=false
   DECISION_RESEARCH_AGENT_ENABLE_EVIDENCE_VERIFICATION=false
   ```

2. For an existing database that must be migrated explicitly outside normal
   startup, run:

   One-line form:

   ```bash
   python scripts/run_identity_migration.py --db "$DECISION_RESEARCH_AGENT_DB_PATH" --backup "$BACKUP_DB"
   ```

   Expanded form:

   ```bash
   python scripts/run_identity_migration.py \
     --db "$DECISION_RESEARCH_AGENT_DB_PATH" \
     --backup "$BACKUP_DB"
   ```

3. To archive pre-v0.1.0 tables, run:

   One-line form:

   ```bash
   python scripts/retire_legacy_database.py --database "$DECISION_RESEARCH_AGENT_DB_PATH" --backup "$BACKUP_DB" --archive "$ARCHIVE_DB"
   ```

   Expanded form:

   ```bash
   python scripts/retire_legacy_database.py \
     --database "$DECISION_RESEARCH_AGENT_DB_PATH" \
     --backup "$BACKUP_DB" \
     --archive "$ARCHIVE_DB"
   ```

   Add `--drop-legacy-tables` only during an operator-reviewed maintenance
   window after the archive has been checked.

4. Start the backend and verify the canonical result path:

   ```bash
   python api/server.py
   python tools/decision_research_agent_tool.py doctor
   python tools/decision_research_agent_tool.py run \
     --query "Research question" \
     --thread-id "demo-thread" \
     --wait
   python tools/decision_research_agent_tool.py result --run-id "$RUN_ID"
   ```

## Rollback

Rollback is an operator procedure, not an automatic runtime fallback.

Before rolling back, stop the backend and restore the application database,
checkpoint database, and output storage together from a consistent backup set.
In short: restore the application database, checkpoint database, and output storage together.
Do not mix a newer application database with an older checkpoint database.

If controlled durable review or evidence verification was enabled, keep both
feature flags false until `doctor`, schema verification, and the durable gate
pass again on the target version.

## Verification

Release readiness is based on commands run in the pull request, not on this
file alone. This document intentionally does not claim that a tag, GitHub
Release, deployment, or final Docker gate has already completed.

Required final checks before tagging:

```bash
python scripts/check_canonical_identity.py --root .
python -m pytest -q
python scripts/durable_hitl_gate_runner.py \
  --output docs/evidence/durable-hitl-gate-report.json
python scripts/real_source_proof.py check-report \
  --report docs/evidence/real-source-proof.json
docker compose config --quiet
docker compose build backend
docker compose up -d mysql backend
ready=0
for attempt in $(seq 1 60); do
  if curl --fail --silent --show-error --max-time 2 \
    http://127.0.0.1:8000/health >/tmp/dra-health.json; then
    ready=1
    break
  fi
  sleep 1
done
if [ "$ready" -ne 1 ]; then
  docker compose logs --no-color backend mysql
  docker compose down -v
  exit 1
fi
curl --fail --silent --show-error --max-time 5 \
  http://127.0.0.1:8000/health
docker compose down -v
```

The readiness loop is bounded to 60 seconds. A timeout prints backend and
MySQL logs, cleans the Compose resources, and fails closed; the final visible
request is the authoritative health result.

## Known Limits

- Single-node SQLite is the documented controlled review and verification
  boundary.
- Evidence verification records operator decisions for exact persisted
  fingerprints; it is not automatic source retrieval or LLM verification.
- LangSmith is privacy-first tracing/evaluation, not a business ledger.
- Application DB = business authority for ResearchRun, EvidenceLedger, review,
  verification, publication, and delivery state.
