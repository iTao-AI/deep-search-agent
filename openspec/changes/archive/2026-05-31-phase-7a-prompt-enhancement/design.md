## Context

当前 `prompt/prompts.yml` 中的 main agent system_prompt 是一段连续文本，缺乏结构化阶段划分。Agent 在执行过程中可能跳过关键步骤（如未收集足够信息就生成报告），导致报告质量不稳定。报告生成也无标准大纲模板。

## Goals / Non-Goals

**Goals:**
- 在 system prompt 中加入四阶段门控工作流，规范 Agent 执行流程
- 定义标准报告大纲模板，确保生成报告结构一致性
- 关键约束从软提示升级为门控检查点，提高执行可靠性

**Non-Goals:**
- 不修改 Agent 代码逻辑（main_agent.py、sub_agents 等）
- 不修改工具实现（markdown_tools、pdf_tools 等）
- 不改变 API 端点或数据流
- 不引入新的 Python 依赖

## Decisions

### 1. 四阶段定义

| 阶段 | 名称 | 目标 | 准入条件 | 产出物 | 门控检查 |
|------|------|------|----------|--------|----------|
| 1 | 需求分析 | 理解用户需求，确定信息来源 | 用户提交任务 | 需求分析报告（包含需要查询的信息点列表） | 建议至少列出 3 个需要查询的信息点（软约束） |
| 2 | 信息收集 | 调用子 Agent 获取所需信息 | 阶段 1 完成 | 信息收集报告（包含各子 Agent 返回的原始数据） | 至少调用 2 个子 Agent，或明确说明为何只需 1 个 |
| 3 | 大纲确认 | 根据收集的信息生成报告大纲 | 阶段 2 完成 | 报告大纲（包含各章节标题和要点） | 大纲必须包含摘要、核心发现、结论建议三个必选章节 |
| 4 | 报告生成 | 按大纲生成完整 Markdown/PDF 报告 | 阶段 3 完成 | 最终报告文件 | 报告字数不少于 1000 字（已有约束，保持） |

### 2. 门控检查实现方式

**选择：Prompt-level gating（非代码级门控）**

理由：
- 最小变更：仅修改 `prompts.yml`，不触碰 Python 代码
- 可逆性强：如果效果不佳，回滚只需恢复 prompt 文本
- 与现有架构兼容：DeepAgents SDK 的 system_prompt 机制天然支持
- 后续可升级：如果 prompt-level 门控不够强，再考虑代码级实现

**门控检查格式：**
在每个阶段结束时，Agent 必须输出明确的阶段完成标记。标记格式遵循 spec 定义的 `【阶段 N 完成】+ 阶段名称 + 关键产出摘要` 模式，具体示例如

```
【阶段 1 完成】需求分析完毕。需要查询的信息点：1)... 2)... 3)...
【阶段 2 完成】信息收集完毕。已调用子 Agent：网络搜索助手、数据库查询助手。
【阶段 3 完成】大纲确认完毕。报告结构：摘要、背景、核心发现、结论建议、参考资料。
【阶段 4 完成】报告生成完毕。
```

### 3. 报告大纲模板结构

```markdown
# [报告标题]

## 摘要
[1-2 段执行摘要，概括核心发现和结论]

## 背景信息
[数据来源、查询范围、分析方法说明]

## 核心发现
[数据分析结果，按重要性排序]

## 结论与建议
[基于发现的结论和可操作建议]

## 参考资料
[引用来源列表]
```

### 4. Prompt 修改范围

**修改 `main_agent.system_prompt`：**
- 在现有 prompt 基础上，新增"【工作流程：四阶段门控】"章节
- 保留现有所有功能描述（子 Agent 介绍、工具说明、路径指令等）
- 新增内容追加到现有 prompt 末尾，不删除任何现有内容
- 报告大纲模板直接内嵌到阶段 4 的 system_prompt 文本中，不新增独立字段或额外 YAML 文件

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| Prompt 过长可能超出 token 限制 | 现有 prompt 约 800 tokens，新增约 500 tokens，总计约 1300 tokens，远低于 Qwen-Max 的限制 |
| Agent 可能忽略门控标记 | 在 prompt 中明确强调"必须"和"禁止"，并用【】标记增强可见性 |
| 四阶段可能增加响应时间 | 门控是流程规范，不增加额外 API 调用，实际执行时间不变 |
| 报告模板过于刚性 | 模板中说明"可根据实际情况调整章节"，保持灵活性 |
